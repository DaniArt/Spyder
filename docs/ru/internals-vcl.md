# Внутренности VCL

Документ описывает опциональную VCL подсистему: как Spyder делает hit-test, строит цепочку и извлекает свойства VCL контролов, включая non-HWND.

## Почему нужен отдельный путь

В VCL могут быть контролы, которые:
- не имеют собственного HWND,
- не отображаются как отдельные UIA элементы,
- требуют работы с VCL runtime структурами для корректного разрешения цепочки.

## UI thread only

VCL операции выполняются строго в UI потоке цели. Spyder обеспечивает это так:
- ставит WH_GETMESSAGE hook в GUI thread цели,
- отправляет thread-message для dispatch,
- обрабатывает его в hook процедуре при прокачке message loop.

## Состав подсистемы

## `VclHook` (native)
- Named Pipe сервер внутри процесса.
- Парсит команды и выполняет их через UI-thread dispatch.
- Защищает risky места: проверки указателей + `__try/__except`.

## `VclBridge`
- Реализует:
  - валидацию объектов,
  - class/name,
  - hit-test,
  - children/path,
  - свойства,
  - сериализацию ответов в JSON.

## `SpyderVclHelper32` (Delphi helper)
- Дает безопасный доступ к VCL API изнутри процесса.
- Используется для:
  - hit-test (`ControlAtPos`, `FindDragTarget`),
  - вычисления rect через координатные преобразования,
  - best-effort чтения некоторых полей.

## Hit-test

1. Resolve VCL root по HWND (best-effort).
2. Hit-test по экранной точке → leaf кандидат.
3. Если leaf — контейнер, делается ограниченный refine:
   - безопасный обход детей,
   - проверка bounds/visibility,
   - ранний stop при нахождении более точного leaf.

## Non-HWND контролы

Для non-HWND:
- поиск идет по VCL дереву (а не по Win32 HWND),
- bounds вычисляются best-effort, через `ClientToScreen` и размеры, либо helper.

## Safety

- Валидация указателей перед чтением.
- `__try/__except` вокруг рискованных мест.
- Ограничения по глубине/узлам обхода.
- Best-effort поведение: при ошибках возвращается частичный результат.
